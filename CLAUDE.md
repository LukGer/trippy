# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repo layout

Bun + Turbo monorepo. Workspaces: `apps/*`, `packages/*`.

- `apps/api` — Cloudflare Workers backend (Hono + tRPC + Drizzle/D1 + better-auth). Deployed via Wrangler.
- `apps/mobile` — Expo / React Native app (expo-router, Tailwind via `uniwind`, tRPC + react-query, `react-native-nitro-fetch`).
- `packages/core` — Everything shared between API and mobile: zod schemas via subpath exports (`@trippy/core/{trips,itinerary,documents}`) and pure TS helpers/constants on the root export (`tripRoles`, `canManageTrip`). No runtime deps on Drizzle, Hono, better-auth, or `@cloudflare/*` — anything env/db/request-bound stays in `apps/api/src/modules/`.
- `packages/tsconfig` — Shared TS configs.

The API is deployed via `wrangler` from `apps/api`; there is no infra-as-code layer in this repo.

## Commands

Run from the repo root unless otherwise noted.

| Task | Command |
| --- | --- |
| Start everything | `bun run dev` (turbo, persistent) |
| API only (Wrangler) | `bun run --cwd apps/api dev` |
| Mobile only (Expo iOS) | `bun run --cwd apps/mobile dev` |
| Typecheck API | `bun run build:api` (alias for `tsc --noEmit` in `apps/api`) |
| Typecheck mobile | `bun run build:app` |
| Lint everything | `bun run lint` (Biome via turbo) |
| Generate D1 migrations | `bun run db:generate` (Drizzle Kit; reads `apps/api/src/{auth,db}/schema.ts`) |
| Apply migrations locally | `bun run db:push` (`wrangler d1 migrations apply trippy-db --local`) |
| Regenerate better-auth schema | `bun run --cwd apps/api auth:generate` |
| Generate Wrangler env types | `bun run --cwd apps/api cf-typegen` |

Tests: none configured. If you add one, wire it through Turbo.

## Architecture notes

### API request flow (`apps/api/src/app.ts`)
Hono app mounts three things on top of CORS:
1. Plain Hono routes registered via `registerPlanUploadRoutes` (R2 presign / upload for plan attachments) and `registerItineraryRoutes` (`POST /api/itinerary/stream`, an SSE-style AI stream).
2. Better-auth: `GET|POST /api/auth/**` is delegated to `createAuth(env).handler`.
3. tRPC at `/trpc/*` via `fetchRequestHandler`; context (`createTRPCContext`) builds per request: `{ auth, db, env, request, session }`.

tRPC procedures: `publicProcedure` and `protectedProcedure` (asserts `ctx.session.user`, throws `UNAUTHORIZED`). Add new procedures by composing on these — don't reach into `initTRPC` again.

### API module convention (`apps/api/src/modules/<feature>/`)
Each feature follows: `<feature>.router.ts` (tRPC procedures) → `<feature>.service.ts` (business logic) → `<feature>.repo.ts` (Drizzle queries) → `<feature>.schemas.ts` (Zod input shapes). Services and repos are **classes** with constructor-injected deps (`class TripsService { constructor(private readonly tripsRepo: TripsRepo) {} }`). Routers map DB rows to DTOs validated against schemas in `@trippy/core/*` before returning.

Composition root is `apps/api/src/modules/services.ts` — `buildServices(db)` constructs every service for a request and is wired onto the tRPC context in `createTRPCContext`. Routers access services via `ctx.services.<feature>` (e.g. `ctx.services.trips.listForUser(...)`) rather than newing them up directly. To add a new module, create the service+repo classes and register them in `buildServices`; do not import a DI framework.

### DB (`apps/api/src/db/`)
Cloudflare D1 (SQLite) via Drizzle. Two schema sources are stitched together: `src/auth/schema.ts` (generated by `@better-auth/cli`) and `src/db/schema.ts` (app tables: `trips`, `tripMembers`, `places`, etc.). Both are passed to `drizzle.config.ts`. Don't hand-edit `src/auth/schema.ts` — regenerate via `auth:generate`. Migrations live in `apps/api/src/db/migrations/`.

### AI itinerary agent (`apps/api/src/modules/itinerary/agent/`)
Vercel AI SDK (`ai`, `@ai-sdk/openai`) with `streamText` + `Output` constrained to `itineraryPlanSchema` from `@trippy/core/itinerary`. Tools: `cover-image.tool.ts` (Unsplash) and `ingest-attachments.tool.ts` (R2-backed PDFs/images/text). Step cap: `stepCountIs(12)`. The route `/api/itinerary/stream` fetches R2 objects by attachment id, then forwards to `createItineraryPlanStreamResponse` which returns a UI-message stream. Mobile parses this stream incrementally in `apps/mobile/src/components/plan-create/itinerary-stream-parse.ts`.

### Mobile structure (`apps/mobile/`)
- `app/` — expo-router file routes. Auth gating via `(authed)` / `(unauthed)` groups; tabs live in `(authed)/(tabs)`.
- `src/components/plan-create/` — the multi-step "create plan" wizard (shell + steps + wizard context + stream transport/parser). When adding wizard steps, update `flow-ids.ts`, `wizard-context.tsx`, and the step component, and respect the existing phase-checklist contract.
- `src/utils/trpc.tsx` — `createTRPCReact<AppRouter>()` is the only way to call the API. The mobile app imports the API router *type* from `@trippy/api/router` (the API package exports `./router`) — this is the type seam that makes tRPC end-to-end typesafe. Don't import API runtime code into mobile.

### Auth
Better-auth with the Expo plugin. Trusted origins built from `CORS_ORIGIN`, `trippy://`, and optionally `exp://` (only when `BETTER_AUTH_ALLOW_EXPO_GO=true` — turn off in production). Google OAuth is enabled only when both `GOOGLE_CLIENT_ID` and `GOOGLE_CLIENT_SECRET` are set. Sessions are read via cookies on the request; mobile sends them through `betterAuthCookieHeaders()`.

### Env / secrets (`apps/api/src/env.ts`)
Bindings: `DB` (D1), `PLAN_UPLOADS` (R2, optional), `BETTER_AUTH_*`, `CORS_ORIGIN`, `GOOGLE_CLIENT_*`, `OPENAI_API_KEY`, `UNSPLASH_ACCESS_KEY`. Secrets go through `wrangler secret put …` (or `.dev.vars` locally). Routes degrade gracefully (e.g. `/api/itinerary/stream` returns 503 when `OPENAI_API_KEY` is missing).
